# Pydantic Models and Naming Conventions

## Context and Problem

### The Issue
FastAPI with Pydantic uses **snake_case** (Python convention), while JavaScript/TypeScript frontends use **camelCase**.

Without proper configuration, this causes:
- ❌ Frontend receives `attachment_file_ids` but expects `attachmentFileIds`
- ❌ State comparisons fail (dirty state detection)
- ❌ Type mismatches and runtime errors
- ❌ Manual conversion needed everywhere

### Example of the Problem
```python
# Backend returns (without BaseAPIModel)
{
  "property_id": 1,
  "photo_file_ids": [1, 2, 3],
  "check_in_sequences": {
    "fr": [{"attachment_file_ids": [1, 2]}]
  }
}

# Frontend expects
{
  "propertyId": 1,
  "photoFileIds": [1, 2, 3],
  "checkInSequences": {
    "fr": [{"attachmentFileIds": [1, 2]}]
  }
}
```

---

## The Solution: BaseAPIModel

### 1. Base Configuration

File: `dev/backend/app/api/base.py`

```python
from pydantic import BaseModel, ConfigDict
from pydantic.alias_generators import to_camel

class BaseAPIModel(BaseModel):
    """
    Base model for all API request/response schemas.

    Features:
    - Automatic camelCase conversion for JSON serialization
    - Accepts both snake_case (Python) and camelCase (Frontend)
    - Ensures consistent naming between frontend and backend
    """
    model_config = ConfigDict(
        alias_generator=to_camel,      # snake_case → camelCase
        populate_by_name=True,          # Accept both formats on input
        from_attributes=True,           # ORM compatibility
    )
```

### 2. FastAPI Global Configuration

File: `dev/backend/app/api/main.py`

```python
app = FastAPI(
    title=settings.app_name,
    response_model_by_alias=True  # ✅ CRITICAL: Force camelCase output globally
)
```

**Why this matters:**
- `response_model_by_alias=True` tells FastAPI to use the aliases generated by `to_camel`
- Without this, the aliases are ignored and snake_case is returned

## How to Structure Response Models

### ❌ WRONG: Using `dict` for nested structures

```python
class MyResponse(BaseAPIModel):
    property_id: int
    check_in_sequences: Optional[dict] = None  # ❌ BAD
    email_fallback: Optional[dict] = None      # ❌ BAD
```

**Problem:** Pydantic only converts **declared fields**, not dict contents.

Result:
```json
{
  "propertyId": 1,  // ✅ Converted
  "checkInSequences": {
    "fr": [{"attachment_file_ids": [1]}]  // ❌ NOT converted
  }
}
```

### ✅ CORRECT: Using typed Pydantic models

```python
class CheckInMessageResponse(BaseAPIModel):
    """Response schema for a single check-in message"""
    order: int
    content: str
    attachment_file_ids: List[int] = Field(default_factory=list)
    # ↑ Will be converted to "attachmentFileIds" automatically

class EmailFallbackResponse(BaseAPIModel):
    """Response schema for email fallback"""
    subject: str
    html: str

class MyResponse(BaseAPIModel):
    property_id: int
    check_in_sequences: Optional[Dict[str, List[CheckInMessageResponse]]] = None  # ✅ GOOD
    email_fallback: Optional[Dict[str, EmailFallbackResponse]] = None              # ✅ GOOD
```

**Result:**
```json
{
  "propertyId": 1,
  "checkInSequences": {
    "fr": [
      {
        "order": 1,
        "content": "Hello",
        "attachmentFileIds": [1, 2, 3]  // ✅ Fully converted!
      }
    ]
  },
  "emailFallback": {
    "fr": {
      "subject": "...",
      "html": "..."
    }
  }
}
```

## Step-by-Step Implementation Guide

### Step 1: Identify All Response Models

Audit all route files in `dev/backend/app/api/routes/`:
- auth.py
- users.py
- properties.py
- reservations.py
- documents.py
- messaging.py
- storage.py
- etc.

### Step 2: Check Each Endpoint Response

For each endpoint:

1. **Identify the response model**
   ```python
   @router.get("/endpoint", response_model=MyResponse)
   async def my_endpoint():
       ...
   ```

2. **Check if model extends BaseAPIModel**
   ```python
   class MyResponse(BaseAPIModel):  # ✅ Good
       ...

   class MyResponse(BaseModel):     # ❌ Bad - should be BaseAPIModel
       ...
   ```

3. **Check for `dict` fields**
   ```python
   class MyResponse(BaseAPIModel):
       data: dict  # ❌ Bad - needs typing
       items: List[dict]  # ❌ Bad - needs typing
   ```

### Step 3: Create Typed Models for Nested Structures

For every `dict` or `List[dict]`, create a proper model:

```python
# Before
class MyResponse(BaseAPIModel):
    items: List[dict]

# After
class Item(BaseAPIModel):
    item_id: int
    name: str
    file_ids: List[int]

class MyResponse(BaseAPIModel):
    items: List[Item]
```

### Step 4: Update Return Statements (if needed)

Most endpoints return raw dicts from CRUD functions. If the CRUD returns:
```python
{
    "property_id": 1,
    "photo_file_ids": [1, 2]
}
```

And you have:
```python
@router.get("/properties/{id}", response_model=PropertyResponse)
async def get_property(id: int):
    result = await crud.get_property(conn, id)
    return result  # ✅ Pydantic will convert automatically
```

**It will work!** Pydantic converts on serialization.

## Common Patterns

### Pattern 1: List Response with Pagination

```python
class PaginationMeta(BaseAPIModel):
    offset: int
    limit: int
    total: int
    has_more: bool

class ItemResponse(BaseAPIModel):
    item_id: int
    file_ids: List[int]
    created_at: str

class ItemsListResponse(BaseAPIModel):
    data: List[ItemResponse]
    pagination: PaginationMeta
```

### Pattern 2: Nested Language-Specific Data

```python
class TranslatedContent(BaseAPIModel):
    order: int
    content: str
    attachment_file_ids: List[int]

class MultiLanguageResponse(BaseAPIModel):
    property_id: int
    sequences: Dict[str, List[TranslatedContent]]  # {"fr": [...], "en": [...]}
```

### Pattern 3: Optional Nested Structures

```python
class DetailedInfo(BaseAPIModel):
    extra_field: str
    metadata: Dict[str, str]

class MyResponse(BaseAPIModel):
    id: int
    name: str
    details: Optional[DetailedInfo] = None  # Can be None
```

## Common Mistakes to Avoid

### ❌ Mistake 1: Forgetting to import Dict
```python
check_in_sequences: Optional[Dict[str, List[Message]]] = None
# NameError: name 'Dict' is not defined
```
**Fix:** `from typing import Dict`

### ❌ Mistake 2: Using BaseModel instead of BaseAPIModel
```python
class MyResponse(BaseModel):  # ❌ Wrong
    ...
```
**Fix:** `class MyResponse(BaseAPIModel):`

### ❌ Mistake 3: Mixing dict and typed models
```python
class MyResponse(BaseAPIModel):
    good_field: List[TypedModel]  # ✅ Good
    bad_field: dict                # ❌ Bad
```
**Fix:** Create a model for `bad_field`

### ❌ Mistake 4: Not testing the actual API response
Assuming it works without checking the actual JSON output.
**Fix:** Always test with curl or Postman

---

## Conclusion

**Golden Rule:** Every API response must use `BaseAPIModel` with fully typed nested structures.

No `dict`, no `List[dict]`, no exceptions.

This ensures consistent camelCase formatting across the entire API surface.
